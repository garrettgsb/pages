❄Flaky Survey

The survey app that lets you change your mind

Description

❄Flaky Survey is a survey app that lets users create surveys and share them with other users through a public link. Users who have completed a survey can later go back and change their answers.

Tech Stack

The app is expected to be designed and built in Ruby on Rails, following the principles covered in "Practical Object Oriented Design in Ruby" by Sandi Metz , but any similar framework such as Django is also acceptable. It can be created as either a single- or multi-page app. The code should be hosted on Github, and it should be relatively easy for another user to clone the repo and launch the app.

App Features

Users can create Surveys that can be shared with other users. Users who have responded to a survey should be able to edit their answers after submitting.

Creating a survey

The Survey creator should be able to add Questions with one of the following answer types:

  • Plain text
  • Multiple choice (select one)
  • Multiple choice (select many)

Sharing a survey

When a survey is created, a public link is generated that lets the survey creator share the survey. Only users who are signed in may respond to the survey.

Responding to a survey

When a user visits a survey belonging to another user, they can fill in answers to each question and submit their results at the end.

If they revisit the public link for a survey that they have already responded to, the survey is populated with their previous answers. They can make any changes that they like, and submit again like normal.

Viewing survey results

Each survey has its own Results Page where the responses for each question are tallied, or in the case of text responses, listed in a way that is easy to browse.

Testing

The app should be unit tested using RSpec or a similar testing framework. It should be fairly easy to see how the app works internally just by looking at the unit tests.

Documentation

Documentation should consist of, at least, a readme file that explains installation, configuration, and basic usage of the app.

Any public interface methods should also be documented. This documentation can live either in the Readme, or externally, such as in a Github wiki or a document generator like Yard. The Readme file should contain links to any external documentation.
